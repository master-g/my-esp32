# 审计整改行动报告（2026-04-09）

## 目标

这份行动报告只处理本次审计里已经坐实的风险，不额外扩 scope。目标也很简单：先把系统从“开发骨架”推进到“能稳定跑的底座”，再继续接业务数据和 UI。

## 总体判断

整改顺序要守一个原则：先修模型，再修单点。

如果状态流、线程边界、显示锁链还保持现在的样子，那么继续给 `Notify`、`Trading`、`device_link` 叠新功能，只会让问题更难定位。当前最值钱的动作，不是“再做一页”，而是先把并发和 I/O 的地基浇实。

## 分阶段行动

### 第一阶段：收口固件状态模型

这一阶段是阻塞项，优先级最高。

#### 动作 1：给运行时状态引入单一写入边界

范围：

- `event_bus`
- `system_state`
- `service_time`
- `service_weather`
- `service_claude`
- `net_manager`
- `service_home`

建议做法：

1. 不再把内部静态快照按裸指针暴露给外部长期读取。
2. 改成“复制快照到调用方 buffer”或者“通过消息队列投递不可变副本”。
3. 明确谁能写状态，谁只能读状态。最理想的形态，是每个 service 只在自己的 task 内修改状态，对外只给副本。

完成标准：

- 不再存在多个 task 直接读写同一个可变静态快照
- `home_service` 聚合时拿到的是一致快照，而不是若干 live pointer

#### 动作 2：把 UI 更新收敛到单一线程

范围：

- `bootstrap`
- `app_manager`
- 所有 `app_*`
- `bsp_display`

建议做法：

1. 不再让 Wi-Fi 事件线程、USB reader task、天气 worker、`esp_timer` 回调直接驱动 app 的 LVGL 更新。
2. 引入 UI command queue，外部线程只投递“需要刷新”的命令。
3. 由 LVGL 所在线程统一消费这些命令并刷新页面。

完成标准：

- app 的 `handle_event()` 不再在任意发布线程里直接跑 LVGL 更新
- 定时器和网络事件线程里不再出现无限等待显示锁的路径

### 第二阶段：拔掉显示链路里的无限等待

这一阶段同样是阻塞项，应和第一阶段并行设计、顺序落地。

#### 动作 3：移除 `portMAX_DELAY` / `UINT32_MAX` 这种无上限等待

范围：

- `app_manager`
- `bsp_display`
- `app_notify`
- 其他直接拿 `bsp_board_lock()` 的路径

建议做法：

1. 显示锁改成有限超时，并在超时时记录明确日志。
2. `lvgl_flush_cb()` 等待 flush done 时增加故障超时和错误状态。
3. 一旦发生超时，优先保系统可恢复，不要把定时器线程和事件线程一起堵死。

完成标准：

- 关键显示路径不存在无限阻塞
- flush 失败时能留下足够的诊断信息

#### 动作 4：补齐 task 创建失败处理

范围：

- `bsp_display`
- `power_runtime`
- `service_weather`
- `service_time`
- `device_link`

建议做法：

1. 检查所有 `xTaskCreatePinnedToCore()` 返回值。
2. 启动失败时直接让初始化链路报错，不要默默继续。
3. `bootstrap_start()` 应该能把这类错误明确冒出来。

完成标准：

- 所有关键后台 task 都有失败检测
- 初始化不会在“半起半不起来”的状态下继续前进

### 第三阶段：修正时间和外部数据语义

#### 动作 5：把 RTC 统一成 UTC 语义

范围：

- `bsp_rtc`
- `service_time`

建议做法：

1. RTC 里只存 UTC 对应的 calendar time。
2. timezone 只影响展示和本地格式化，不影响 RTC 持久化语义。
3. timezone 变更后，当前系统显示文本更新即可，不要让已存 RTC 数据重新解释成另一套 epoch。

完成标准：

- 断网重启后，RTC 恢复结果不依赖当前 `TZ`
- timezone 修改不会让 RTC 恢复出的 epoch 漂移

#### 动作 6：把天气响应解析改成结构化解析

范围：

- `weather_client`

建议做法：

1. 用现有 `cJSON` 组件解析 `current.temperature_2m`、`current.weather_code`、`current.is_day`。
2. 对响应体大小增加显式上限和专门错误码，区分“响应过大”和“结构不匹配”。
3. 保留旧快照时，要把失败原因落日志，不要只给一个笼统的 `ESP_ERR_INVALID_RESPONSE`。

完成标准：

- 上游字段顺序变化不会影响解析
- 响应过长、字段缺失、类型不匹配能被明确区分

### 第四阶段：加固 Host 侧持久化和串口边界

#### 动作 7：把 `state.json` 改成原子写入

范围：

- `tools/esp32dash/src/agent.rs`

建议做法：

1. 先写临时文件。
2. `fsync` 后再 rename 覆盖正式文件。
3. 读取失败时把“文件损坏”单独记日志，不要直接静默当作首次启动。

完成标准：

- agent 异常退出不会留下半截状态文件
- 状态文件损坏时有明确可见的错误信息

#### 动作 8：给串口读缓冲和 worker 通道加护栏

范围：

- `tools/esp32dash/src/device.rs`
- `tools/esp32dash/src/compat.rs`

建议做法：

1. 给 `read_buf` 增加最大行长，超限后丢弃当前帧并记录日志。
2. `send_snapshot()` 不再吞掉 `send()` 错误，至少要打日志并反映到状态里。
3. `ESP32DASH_SERIAL_BAUD` 解析失败时输出警告，不要静默退回默认值。

完成标准：

- 异常串口帧不会导致 host 内存持续增长
- worker 失效和配置错误能被操作方看见

### 第五阶段：补回归和放行条件

这一步不解决新问题，只负责确认前面的修补真的站住了。

#### 动作 9：补最小验证闭环

范围：

- 固件构建
- Host 测试
- 手工/脚本化回归

建议做法：

1. 在可用的 ESP-IDF 环境里执行 `idf.py build`。
2. 保持 `tools/esp32dash` 现有 Rust 测试常态通过。
3. 增加几类最低限度的回归：
   - 高频事件下页面刷新不死锁
   - Wi-Fi 断开 / 重连后系统仍可交互
   - 串口收到超长无换行数据时 host 不膨胀
   - timezone 修改后 RTC 恢复语义不漂移

完成标准：

- 固件能在正式 IDF 环境稳定构建
- Host 现有测试持续通过
- 上面四类异常路径至少各有一轮明确回归记录

## 建议执行顺序

按顺序做，不要交叉得太散：

1. 第一阶段：固件状态模型
2. 第二阶段：显示链路超时和 task 创建失败处理
3. 第三阶段：RTC 与天气解析
4. 第四阶段：Host 持久化与串口边界
5. 第五阶段：构建与回归

原因很简单。前两阶段是在修地基，后三阶段是在加护栏。地基没修好，后面做得再认真，也只是在不稳定模型上继续堆功能。

## 放行标准

在进入下一轮真实业务接线之前，至少应该满足这几条：

1. 固件 UI 更新只从单一线程进入 LVGL。
2. 显示锁和 flush 路径不再有无限等待。
3. 关键后台 task 创建失败会让初始化直接失败。
4. RTC 采用 UTC 语义，timezone 只影响展示。
5. Host `state.json` 是原子写入。
6. 串口读缓冲有上限，worker 异常不会被静默吞掉。
7. `idf.py build` 和 Rust 现有测试都能稳定通过。

## 备注

这份行动报告故意没有把范围扩到“加 CI”“重做所有 service”“完成全部页面”，因为那会把优先级打散。眼下最要紧的，是先把已经暴露出来的稳定性问题补平。

只要这几项收口做完，后面继续接 `Claude`、`Weather`、`Market` 的真实数据，代价会小很多，问题也会更好查。
