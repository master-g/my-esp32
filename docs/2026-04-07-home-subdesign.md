# Home 页面子设计方案

## 1. 目的

本文档细化主设计文档中的 `Home` 页面，目标是把下面这些问题固定下来：

- 首页到底显示哪些信息
- `time_service`、`weather_service`、联网状态如何汇聚到一个快照
- 亮屏、待机、低功耗下首页如何退化
- 手动刷新和错误态如何表现

本文档是 [2025-04-06-esp32-dashboard-design.md](./2025-04-06-esp32-dashboard-design.md) 中 `Home` 页面的补充说明。

## 2. 设计边界

### 2.1 覆盖范围

- `Home` 页面状态和布局
- `time_service`、`weather_service`、联网状态在首页上的聚合
- 首页的低功耗显示策略
- 首页的手动刷新和错误态

### 2.2 非覆盖范围

- 完整天气设置页
- 多城市切换
- 自定义背景图编辑器
- 深度桌面小组件系统

## 3. 页面职责

`Home` 是系统默认首页，职责是提供“一眼可读”的基础状态：

- 当前时间
- 日期和星期
- 当前联网状态
- 当前天气摘要
- 一个轻量的系统角标区

它不是：

- 复杂交互页
- 高频动画页
- 网络调试页

## 4. 服务聚合模型

首页只消费聚合快照，不直接访问底层服务。聚合对象定义为：

```c
typedef struct {
    bool rtc_valid;
    bool ntp_synced;
    bool wifi_connected;
    bool weather_available;
    bool weather_stale;
    uint32_t updated_at_epoch_s;
    char time_hhmm[6];
    char date_text[24];
    char weekday_text[12];
    char weather_text[24];
    char city_text[24];
    int16_t temperature_c_tenths;
    uint8_t weather_icon_id;
    bool claude_unread;
} home_snapshot_t;
```

约束：

- `Home` 页面不关心 `time_service` 和 `weather_service` 的内部缓存结构
- 页面只消费格式化好的字符串和少量枚举/标志位
- 所有格式化逻辑尽量下沉到 service 或聚合层

## 5. 时间模型

### 5.1 时间来源优先级

首页时间显示优先级固定为：

1. `RTC + 已知时区`
2. `NTP 同步后的系统时间`
3. `RTC 未校准占位态`

设计意图：

- 重启后即使还没联网，也尽量先显示 RTC 时间
- NTP 同步成功后更新 RTC
- 如果 RTC 无效，则页面必须明确显示“时间未同步”，而不是静默显示 `00:00`

### 5.2 时间格式

v1 固定使用：

- 时间：`HH:MM`
- 日期：`M月D日`
- 星期：`周一` 到 `周日`

首页不显示秒数。

原因：

- 秒级跳动会增加无意义重绘
- 172x640 竖屏上，分钟级显示更稳定、更可读

## 6. 天气模型

### 6.1 天气数据来源

首页天气只使用 `weather_service` 的缓存结果，不直接发请求。

首页仅展示：

- 温度
- 图标
- 城市名
- 简短天气文本

### 6.2 天气状态

天气显示分为四种状态：

- `LIVE`：最近一次更新成功且未过期
- `STALE`：存在旧缓存，但已过期
- `UNAVAILABLE`：从未获取成功
- `REFRESHING`：手动刷新已触发，等待服务返回

首页页面不需要额外维护复杂状态机，而是由以下字段推导：

- `weather_available`
- `weather_stale`
- `wifi_connected`

### 6.3 展示策略

- `LIVE`：正常显示图标、温度、城市
- `STALE`：正常显示，但在角标区标记 `cached`
- `UNAVAILABLE`：显示 `--` 和默认天气图标
- `REFRESHING`：保持旧数据或占位，不做整页闪烁

## 7. 页面状态与低功耗行为

### 7.1 页面状态

`Home` 页面只维护两个显示层状态：

- `HOME_VIEW_NORMAL`
- `HOME_VIEW_DEGRADED`

进入 `DEGRADED` 的条件：

- 时间源不可用
- 天气不可用
- 或页面处于低功耗亮度策略

### 7.2 低功耗策略

首页配合系统 `ACTIVE / DIM / SLEEP` 状态工作：

- `ACTIVE`：显示完整首页
- `DIM`：保留时间、日期和最小角标，弱化天气信息
- `SLEEP`：停止页面刷新，只保留系统唤醒入口

约束：

- `DIM` 不重新布局整页，只降低亮度并裁剪次要信息
- 页面不主动阻止系统进入 `DIM` 或 `SLEEP`

## 8. 交互设计

### 8.1 支持的交互

首页只支持以下轻量交互：

- 左右边缘翻页
- 点击天气区域触发手动刷新
- 点击时间区域可选触发校时状态提示

### 8.2 不支持的交互

- 长按进入复杂设置
- 首页内的左右滑动手势
- 复杂卡片展开/折叠

### 8.3 手动刷新

点击天气区域后：

1. 页面发出刷新请求事件
2. `weather_service` 决定是否立即刷新
3. 页面进入短暂 `REFRESHING` 视觉状态
4. 结果成功或失败后回到常规显示

约束：

- 手动刷新必须有最小节流，默认 `60 s`
- 节流在 service 层处理，不在 UI 层硬编码时间

## 9. 布局细化

### 9.1 布局区域

页面分成四个区域：

1. 顶部角标区：Wi-Fi、Claude 未读、小状态
2. 中央时间区：大号时钟
3. 日期区：日期和星期
4. 底部天气区：图标、温度、城市

### 9.2 布局原则

- 时间必须是视觉焦点
- 日期和天气属于次要信息
- 角标区不应干扰主时间阅读
- 所有文本在最差情况下都必须允许缩略或裁剪，而不是溢出

## 10. 对外接口

首页不直接消费多个 service，而是通过聚合接口：

```c
const home_snapshot_t *home_service_get_snapshot(void);
void home_service_request_weather_refresh(void);
bool home_service_can_refresh_weather(void);
```

设计原则：

- `Home` 页面只拉聚合快照
- 刷新节流与权限判断在聚合层或 service 层完成
- 页面不直接读取 Wi-Fi、RTC、天气底层对象

## 11. 错误和退化处理

### 11.1 时间不可用

如果时间不可用：

- 大号时钟显示 `--:--`
- 日期区显示 `Time not synced`
- 允许系统联网或后续 RTC 恢复后自动更新

### 11.2 天气不可用

如果天气不可用：

- 不阻塞首页显示
- 底部区域显示默认图标和 `--`
- 若存在旧缓存，显示 `cached`

### 11.3 网络断开

如果 Wi-Fi 断开：

- 顶部角标区显示离线
- 时间照常显示
- 天气继续显示旧缓存或占位

## 12. 验收用例

### 12.1 正常显示

- RTC 和天气缓存都有效时，首页完整显示
- 每分钟边界更新时间字符串

### 12.2 退化显示

- 没有天气时，首页仍然能稳定显示时间和日期
- 时间未同步时，首页明确显示占位态

### 12.3 手动刷新

- 点击天气区域后能触发刷新请求
- 在节流窗口内重复点击不会无限发请求

### 12.4 低功耗

- `DIM` 时首页弱化次要信息但不丢主时间
- `SLEEP` 唤醒后首页能从快照恢复

## 13. 假设

- v1 使用单城市天气模型
- 时区配置由系统层提供，不在首页单独维护
- 首页视觉主题遵循主设计，不在本文档中扩展皮肤变体
